# 설명 문서 (최소 시나리오 기준)

이 문서는 `minimal_scenario.c`의 **최소 구현**을 기준으로 현재 로직을 설명하며,  
첨부된 4페이지(프로토콜 정의)와 9페이지(예시)를 반영해 **프레임 구조를 명시**합니다.

## 1) tx_buf가 128인 이유
- 현재 코드는 **최소 시나리오**이므로 고정 크기 `128`을 **임시 상수**로 사용했습니다.
- 사양서(4페이지, 9페이지 예시) 기준으로 프레임은 **바이너리 구조**이며 길이는 `MsgLength`로 결정됩니다.
- 9페이지 예시를 보면 `MsgLength` 값이 `0x05`, `0x06`, `0x09`처럼 작습니다. 즉 **정상 프레임은 짧은 편**입니다.
- 따라서 `tx_buf=128`은 **과도하게 큰 안전 버퍼**일 뿐이며, 실제 구현에서는 **최대 MsgLength**를 기준으로
  버퍼 크기를 산정하는 것이 맞습니다. (예: `헤더 5바이트 + PID 2바이트 + Reserved 2바이트 + Data n바이트 + CRC 2바이트`)

## 2) addr을 받아서 버퍼에 넣는 방식
- 사양서 4페이지 구조에 따르면 **Address는 프레임의 첫 바이트(오프셋 0)** 입니다.
- 따라서 실제 구현에서는 `tx_buf[0] = addr;` 형태의 **바이너리 1바이트 삽입**이 되어야 합니다.
- 현재 `minimal_scenario.c`는 최소 시나리오로 ASCII 프레임을 사용하기 때문에 문자열에 삽입하는 방식입니다.
  - 예: `"ADDR=1;CMD=1;PID=READ_X\r\n"`
- 결론적으로, **사양서 기준 구현에서는 Address를 텍스트가 아닌 1바이트 필드로 넣어야 합니다.**

## 3) ACK, MsgLength, Reserved는 어디에 있는가?
**사양서 기준 위치**
- ACK: 오프셋 2 (1바이트)
- MsgLength: 오프셋 3 (1바이트)
- Reserved: 오프셋 7~8 (2바이트)

**현재 최소 구현 상태**
- `minimal_scenario.c`는 ASCII 기반 임시 포맷이라 **ACK/MsgLength/Reserved가 실제 필드로 존재하지 않습니다.**
- 사양서 기준 구현 시, 아래 순서로 바이너리 프레임을 채워야 합니다.
  1. `Address` (오프셋 0)
  2. `Device ID` (오프셋 1)
  3. `Ack` (오프셋 2)
  4. `MsgLength` (오프셋 3)
  5. `Cmd` (오프셋 4)
  6. `PID` (오프셋 5~6, MSB/LSB)
  7. `Reserved` (오프셋 7~8)
  8. `Data` (오프셋 9~9+n-1)
  9. `CRC` (오프셋 9+n, 9+n+1)

## 4) 사양서 기반 정리 (핵심 포인트)
4페이지 프로토콜 및 9페이지 예시 기준으로 다음이 확인됩니다.
- `Cmd`는 1~4가 존재하며, **Master→Device 전송은 Cmd 1(읽기), Cmd 3(쓰기)** 사용.
- `PID`는 2바이트 (MSB/LSB)로 표기.
- `Reserved`는 2바이트.
- `CRC`는 2바이트이며 프레임 마지막에 위치.
- `MsgLength`는 프레임 길이를 나타내며 예시 값 `0x05`, `0x06`, `0x09`가 사용됨.

**아직 확정이 필요한 항목**
- `MsgLength`의 정확한 계산 규칙 (헤더 포함 여부 등)
- `ACK` 값의 의미/규칙 (예시상 0x00 요청, 0x01 응답)
- CRC 계산 방식 (다항식, 초기값 등)
