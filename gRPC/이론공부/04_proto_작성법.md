# proto 작성법 및 주의 사항

이 문서는 `.proto` 규약의 핵심 작성법과 주의점을 정리합니다.
작성 언어와 관계없이 동일한 구조와 규약을 따르며, 이 파일이 서버와 클라이언트의 계약서 역할을 합니다.

## 1. 기본 구조

- `syntax = "proto3";` 선언
- `package` 선언(권장)
- `service`와 `rpc` 정의
- `message` 정의

## 2. 작성 방법(기초)

- 파일 맨 위에 `syntax`를 먼저 선언합니다.
- `package`는 이름 충돌을 줄이기 위해 권장합니다.
- 서비스는 `service` 블록 안에 `rpc`로 정의합니다.
- 메시지는 `message`로 정의하고 각 필드에 고유한 번호를 부여합니다.
- 필드 번호는 1부터 시작하며, 한 번 사용한 번호는 재사용하지 않는 것이 안전합니다.

## 3. 예시(요청/응답)

아래 예시는 가장 기본적인 단일 요청/응답 RPC입니다.

```proto
syntax = "proto3";

package helloworld;

service Greeter {
  rpc SayHello (HelloRequest) returns (HelloReply);
}

message HelloRequest {
  string name = 1;
}

message HelloReply {
  string message = 1;
}
```

## 4. 더 알아두면 좋은 규칙

- 필드 번호(`= 1`, `= 2` 등)는 변경에 주의해야 합니다.
- 기존 필드 번호 재사용은 호환성 문제를 일으킬 수 있습니다.
- `proto3`는 기본값 처리 방식이 `proto2`와 다릅니다.
- 타입을 바꿀 때는 호환성(Forward/Backward)을 고려해야 합니다.
- `optional`은 필요한 경우에만 사용하고, 기본값 처리 차이를 이해해야 합니다.

## 5. 언어와 무관한 규약 여부

`.proto`는 언어에 독립적이며, 서버/클라이언트가 어떤 언어로 작성되었든 동일한 구조와 규약을 따릅니다.

## 6. 서버/클라이언트 공유 규칙

- 서버와 클라이언트는 동일한 스키마를 공유해야 합니다.
- 파일 위치는 달라도 되지만, 내용은 동일해야 합니다.

## 7. 용어 정리

- gRPC는 플러그인이 아니라 통신 규약과 이를 구현한 라이브러리(프레임워크)입니다.
- “플러그인”이라는 말은 보통 `protoc`에 붙는 코드 생성 플러그인을 의미합니다.
